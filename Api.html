核心概念

Fetch API 是什麼？

Fetch API 是瀏覽器內建的現代 HTTP 客戶端，取代舊式的 XMLHttpRequest。它以 Promise 為基礎，讓非同步請求的寫法更簡潔。

📦
Request 物件

描述「我要發出什麼樣的請求」，包含 URL、method、headers、body 等設定。

📬
Response 物件

描述「伺服器回傳了什麼」，包含 status、headers，以及讀取 body 的方法。

🔑
Headers 物件

用來操作 HTTP 標頭的容器，支援 get/set/append/has 等方法。

基本用法

GET 請求

JavaScript 
// 最簡單的 GET 請求
const response = await fetch('https://localhost:7001/api/exercises');

// response 是 Response 物件，需要再解析 body
const data = await response.json();   // 解析 JSON
// const text = await response.text();  解析純文字
console.log(data);

POST 請求（帶 JSON Body）

JavaScript 
const response = await fetch('https://localhost:7001/api/exercises', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: '深蹲',
    category: '腿部'
  })
});

if (!response.ok) {
  // response.ok 為 true 代表 status 在 200–299 之間
  throw new Error(`HTTP error! status: ${response.status}`);
}
const created = await response.json();

使用 Request 物件（進階）

JavaScript 
// 先建立 Request 物件，再傳給 fetch
const req = new Request('https://localhost:7001/api/exercises', {
  method: 'GET',
  headers: new Headers({ 'Accept': 'application/json' }),
  mode: 'cors'          // 跨來源請求（預設值）
});

const response = await fetch(req);
Response 物件重要屬性
屬性 / 方法 型別 說明
.status number HTTP 狀態碼，例如 200、404、500
.ok boolean status 在 200–299 之間則為 true
.headers Headers 回應標頭物件
.url string 最終請求的 URL（可能經過重新導向）
.json() Promise 將 body 解析為 JSON 物件
.text() Promise 將 body 解析為純文字字串
.blob() Promise 將 body 解析為 Blob（二進位）
⚠️ 注意

fetch() 只有在「網路錯誤」或請求被中斷時才會 reject Promise。
就算伺服器回傳 404 或 500，Promise 仍然是 resolve。必須手動檢查 response.ok。

同源政策 & CORS

為什麼需要 CORS？

瀏覽器預設有同源政策（Same-Origin Policy）：網頁只能向「同一來源」發出完整的 HTTP 請求。來源（Origin）由三部分組成：

https://localhost:5500

任何一個不同 → 不同來源 → 瀏覽器阻擋

CORS（跨來源資源共享）是一種讓伺服器「授權」特定外來 Origin 的機制。後端透過回應標頭告訴瀏覽器：「這個來源我允許存取。」

CORS 回應標頭
標頭 作用
Access-Control-Allow-Origin 允許的 Origin，可指定特定來源或 *（萬用，但不能與 credentials 同用）
Access-Control-Allow-Methods 允許的 HTTP 方法，如 GET, POST, PUT, DELETE
Access-Control-Allow-Headers 允許的請求標頭，如 Content-Type, Authorization
Access-Control-Max-Age 預檢結果快取秒數（避免每次都預檢）
Access-Control-Allow-Credentials 是否允許攜帶 Cookie，設為 true 時 Origin 不能用 *
Origin 請求標頭

Origin 標頭的語意

瀏覽器在發出跨來源請求時，會自動在請求中附上 Origin 標頭，告訴伺服器「這個請求從哪裡來」。

HTTP Request Headers 
Origin: https://localhost:5500
Host: localhost:7001
Content-Type: application/json
💡 重點

Origin 標頭只包含「協定 + 網域 + 埠號」，不包含路徑。它由瀏覽器自動加入，前端無法用 JS 覆寫此標頭（受瀏覽器安全限制保護）。

Origin vs Referer vs Host
標頭 包含內容 由誰設定 可否偽造（前端）
Origin 協定 + 網域 + 埠 瀏覽器（跨來源時） ❌ 不可
Referer 完整 URL（含路徑） 瀏覽器 ❌ 不可（可被省略）
Host 目標伺服器的網域 + 埠 瀏覽器 ❌ 不可

伺服器如何利用 Origin？

ASP.NET Core 的 CORS 中介軟體會比對請求的 Origin 標頭與政策中設定的允許清單。若符合，才加入對應的 Access-Control-Allow-Origin 回應標頭。

請求帶 Origin: http://localhost:5500
──▶
CORS 中介軟體
在允許清單？
是 → 加入 Access-Control-Allow-Origin 回應
不在清單？
否 → 不加回應標頭，瀏覽器阻擋
特殊情況

何時不會帶 Origin？

🔗
同源請求

前後端同一 Origin，瀏覽器通常不加 Origin 標頭。

📄
直接輸入 URL

在瀏覽器網址列直接輸入，不帶 Origin。

🛠️
Postman / curl

不是瀏覽器環境，不自動帶 Origin，CORS 限制不適用。

⚠️ CORS 只存在於瀏覽器

CORS 是瀏覽器的安全機制，不是伺服器防護。後端 API 本身沒有被「保護」，任何非瀏覽器工具（curl、Postman、伺服器對伺服器）都可以正常存取，不受 CORS 限制。

ASP.NET Core 設定

Program.cs CORS 設定

在 ASP.NET Core 中，CORS 透過中介軟體（Middleware）與政策（Policy）兩步驟設定。

完整範例
Program.cs 
var builder = WebApplication.CreateBuilder(args);

// ① 定義 CORS 政策名稱（方便後續引用）
const string MyAllowedOrigins = "_myAllowedOrigins";

// ② 註冊 CORS 服務並定義政策
builder.Services.AddCors(options =>
{
    options.AddPolicy(name: MyAllowedOrigins,
        policy =>
        {
            policy
                // 指定允許的 Origin（HTTP Origin 標頭的值）
                .WithOrigins(
                    "http://localhost:5500",    // Live Server
                    "http://127.0.0.1:5500"      // 同上，不同寫法
                )
                // 允許的 HTTP 方法
                .WithMethods("GET", "POST", "PUT", "DELETE")
                // 允許前端帶的請求標頭
                .WithHeaders("Content-Type", "Authorization")
                // 若需要 Cookie，加此行（Origin 就不能用 AllowAnyOrigin）
                // .AllowCredentials()
                ;
        });
});

builder.Services.AddControllers();

var app = builder.Build();

// ③ 加入 CORS 中介軟體（順序很重要！）
app.UseHttpsRedirection();
app.UseCors(MyAllowedOrigins);   // ← 在 UseAuthorization 之前
app.UseAuthorization();

app.MapControllers();
app.Run();
🚨 中介軟體順序

UseCors() 必須在 UseRouting() 之後、UseAuthorization() 之前。順序錯誤會導致 CORS 標頭沒有正確加入回應。

常用政策寫法

開發階段：允許所有來源（快速但不安全）

Program.cs（開發用） 
policy
    .AllowAnyOrigin()      // 等同於 Access-Control-Allow-Origin: *
    .AllowAnyMethod()
    .AllowAnyHeader();

Attribute 方式（套用到特定 Controller）

ExerciseController.cs 
[ApiController]
[Route("api/[controller]")]
[EnableCors("_myAllowedOrigins")]   // 套用特定政策
public class ExerciseController : ControllerBase
{
    // ...
}
💡 WithOrigins 與 Origin 標頭的關係

前端請求帶的 Origin: http://localhost:5500 標頭，會被 CORS 中介軟體與 WithOrigins() 中的清單比對。完全符合（含協定與埠號）才算通過，大小寫不分，但斜線結尾有差異。

常見錯誤排查
瀏覽器錯誤訊息 可能原因 解法
No 'Access-Control-Allow-Origin' header 後端沒有 UseCors，或 Origin 不在清單中 確認 Program.cs 有加 UseCors()，且來源 URL 完全符合
Method not allowed by CORS WithMethods 未包含該 HTTP 方法 在政策中加入對應方法，如 .WithMethods("DELETE")
CORS with credentials 錯誤 使用了 AllowAnyOrigin 又開啟 credentials 改用 WithOrigins() 指定具體來源
Postman 可以但瀏覽器不行 正常！CORS 只作用於瀏覽器 確認 Program.cs 的 CORS 設定後再用瀏覽器測試
